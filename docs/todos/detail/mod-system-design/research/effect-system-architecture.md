# Effect System Architecture Research

Supports the Protocol & Hazard System Design todo. Covers the current effect system
(`src/effect/`), the planned new effect system (`src/new_effect/` — todo #2), and how
protocols should target the new system.

---

## Part 1 — Current Effect System (`src/effect/`)

### Core Data Model

Two components live on game entities (bolt, breaker, cell, wall) that carry effects:

**`BoundEffects`** (`Vec<(String, EffectNode)>`)
- Permanent effect trees keyed by chip name.
- Written at chip equip time (via `dispatch_chip_effects`) or at runtime via `On` node
  resolution. Never consumed — re-fires every time the trigger fires.

**`StagedEffects`** (`Vec<(String, EffectNode)>`)
- One-shot armed entries consumed when their trigger fires.
- Populated by `BoundEffects` evaluation when a `When` node's children are not bare `Do`
  nodes (they get staged for the next-level trigger). Also populated by `Until` desugaring.

The two components are always inserted as a pair (`ensure_effect_components`).

### `EffectNode` Tree Shape (current)

```
RootEffect::On { target, then: Vec<EffectNode> }
  EffectNode::When { trigger, then }    -- gate, never consumed
  EffectNode::Do(EffectKind)           -- terminal, fires immediately
  EffectNode::Once(Vec<EffectNode>)    -- first-match gate, consumes itself
  EffectNode::On { target, permanent, then } -- runtime redirect to another entity
  EffectNode::Until { trigger, then }  -- fires now, reverses when trigger fires
  EffectNode::Reverse { effects, chains } -- internal, generated by Until desugaring
```

`EffectKind` has 26 variants covering all current effects (shockwave, speed boost, damage
boost, piercing, size boost, bump force, attraction, chain lightning, pulse, anchor, etc.).

### Trigger Vocabulary (current)

All triggers are variants of the `Trigger` enum:

**Global triggers** (fire on ALL entities with `BoundEffects`):
`PerfectBump`, `EarlyBump`, `LateBump`, `Bump`, `BumpWhiff`, `NoBump`,
`Impact(ImpactTarget)`, `Death`, `BoltLost`, `CellDestroyed`, `NodeStart`, `NodeEnd`,
`NodeTimerThreshold(f32)`, `TimeExpires(f32)`

**Targeted triggers** (fire only on the specific participating entity):
`PerfectBumped`, `EarlyBumped`, `LateBumped`, `Bumped` (on bolt),
`Impacted(ImpactTarget)` (on both collision participants), `Died` (on victim entity)

`ImpactTarget` variants: `Cell`, `Bolt`, `Wall`, `Breaker`

### Bridge System Pattern

Every trigger type has a dedicated bridge system in `src/effect/triggers/`. Each bridge:

1. Reads a game message (e.g., `BoltImpactCell`, `BumpPerformed`, `CellDestroyedAt`)
2. Builds a `TriggerContext { bolt, breaker, cell, wall }` with participant entities
3. For global triggers: iterates ALL entities with `BoundEffects` and evaluates each
4. For targeted triggers: queries only the specific participant entities

All bridge systems run in `FixedUpdate` in the `EffectSystems::Bridge` system set,
with `run_if(in_state(NodeState::Playing))` guards. The `node_start` bridge is an
exception: it runs in `OnEnter(NodeState::Playing)`.

**Evaluation call chain:**
```
bridge_<trigger>
  → evaluate_bound_effects(trigger, entity, bound, staged, commands, context)
    walks BoundEffects: When nodes whose trigger matches → stage children into StagedEffects
    OR fires bare Do children immediately via commands.fire_effect
  → evaluate_staged_effects(trigger, entity, staged, commands, context)
    walks StagedEffects: consumed entries fire their Do children, or arm nested On/When nodes
```

Note: The current code walks `BoundEffects` first, then `StagedEffects`. The new system
flips this order (StagedEffects first) to prevent a single trigger from both arming and
consuming a nested `When` in the same dispatch — see Part 2.

### How Chips Install Effects

`dispatch_chip_effects` system (`src/chips/systems/dispatch_chip_effects/`):

1. Reads `ChipSelected` message
2. Looks up chip definition in `ChipCatalog` (built from `.chip.ron` and `.evolution.ron` files)
3. For `Target::Breaker` root effects: resolves breaker entities, calls `dispatch_children`
   - `Do` children fire immediately via `commands.fire_effect`
   - `On { target }` children recurse to inner entity
   - All other children transfer to entity's `BoundEffects` via `commands.transfer_effect`
4. For all other targets (Bolt, Cell, etc.): wraps effect in `When(NodeStart, On(target, permanent: true, ...))` and pushes to breaker's `BoundEffects` — deferred dispatch fires at node start

**Chip definition shape (RON):**
```ron
// From a .chip.ron file
(
    name: "Speed Chip",
    max_taken: 3,
    common: (
        prefix: "Basic",
        effects: [
            On(target: Bolt, then: [
                When(trigger: Impacted(Cell), then: [Do(SpeedBoost(multiplier: 1.2))])
            ])
        ]
    ),
)
```

Each `ChipTemplate` expands into `ChipDefinition` structs per rarity slot. Evolutions follow
the same `effects: Vec<RootEffect>` shape but are loaded from `.evolution.ron`.

### How `Until` Works (Current)

`desugar_until` system in `FixedUpdate` during `NodeState::Playing`:

1. Extracts all `Until { trigger, then }` nodes from both `BoundEffects` and `StagedEffects`
2. For each `Until`:
   - Fires all bare `Do` children immediately
   - Pushes non-`Do` children to `BoundEffects` (these are chains that activate during the duration)
   - Inserts `When { trigger, Reverse { effects, chains } }` into `StagedEffects` (the reversal)
3. When the reversal trigger fires: `Reverse` node calls `reverse_effect` for each effect,
   and `RemoveChainsCommand` removes the installed chains from `BoundEffects`

### `EffectCommandsExt` (current API)

```rust
commands.fire_effect(entity, effect, source_chip)
commands.reverse_effect(entity, effect, source_chip)
commands.transfer_effect(entity, chip_name, children, permanent, context)
commands.push_bound_effects(entity, effects)
commands.dispatch_initial_effects(effects, source_chip)
```

All of these queue Bevy `Command` structs that apply to `world` at end-of-frame.

### Source Attribution

`EffectSourceChip(Option<String>)` is a component on spawned effect entities (shockwave
rings, pulse rings, etc.). The `chip_name` string key in `BoundEffects` / `StagedEffects`
entries carries the originating chip name throughout the tree walk. This is the only
identity tracking the current system has — there is no cleanup mechanism to remove all
effects from a chip by name.

## Part 2 — Planned New Effect System (todo #2: `src/new_effect/`)

Source files: `docs/todos/detail/effect-desugaring-node-running-trigger/`

### What Changes

The new system is a clean-room implementation. The current system stays in `src/effect/`
and is swapped out in phase 6 after the new system is fully validated.

### New Storage Shape

**`BoundEffects` becomes a `HashMap`-keyed structure:**
```rust
struct BoundEffects {
    triggers: HashMap<Trigger, Vec<BoundEntry>>,   // event-driven entries
    conditions: HashMap<Condition, Vec<BoundEntry>>, // state-driven entries
    sources: HashMap<SourceId, Vec<BoundKey>>,     // reverse index for removal
}
struct BoundEntry { source: SourceId, tree: ValidTree }
type SourceId = String; // chip name, protocol name, etc.
```

**`StagedEffects` also becomes `HashMap`-keyed:**
```rust
struct StagedEffects {
    entries: HashMap<Trigger, Vec<StagedEntry>>,  // consumed when triggered
}
```

**`SpawnedRegistry` (new `Resource`):**
```rust
struct SpawnedRegistry {
    entries: HashMap<EntityType, Vec<SpawnedEntry>>, // applied to newly spawned entities
}
```

### New `ValidTree` Shape

```
ValidDef { stamp_target: RouteTarget, tree: ValidTree }

ValidTree:
  Fire(EffectType)
  When(Trigger, Box<ValidTree>)
  Once(Trigger, Box<ValidTree>)      -- self-removes after first match
  During(Condition, ValidScopedTree) -- state-scoped, stays in BoundEffects.conditions
  Until(Trigger, ValidScopedTree)    -- event-scoped, self-removes after reversal
  Spawned(EntityType, Box<ValidTree>)
  On(ParticipantTarget, ValidTerminal)  -- non-This target redirect only

ValidScopedTree:
  Fire(ReversibleEffectType)   -- reversible effects only in direct position
  When(Trigger, Box<ValidTree>) -- nested When relaxes to any effect
  On(ParticipantTarget, ValidScopedTerminal)

ValidTerminal:
  Fire(EffectType)
  Stamp(Box<ValidTree>)    -- permanent add to target's BoundEffects
  Transfer(Box<ValidTree>) -- one-shot add to target's StagedEffects
  Reverse(ReversibleEffectType)  -- internal only, generated by During/Until
```

### Key New Primitives

**`Route`** (replaces `RootEffect::On`): definition-level wrapper. Sets `This` for the
subtree and tells the dispatch system which entity type to route onto. Required at the root
of every `effects:[]` entry. Variants include `Bolt`, `Breaker`, `ActiveBolts`, `EveryBolt`
(desugars to `ActiveBolts + Spawned(Bolt)`), etc.

**`Stamp`** (new terminal, replaces non-permanent `On` usage): permanently adds an inner
tree to a target entity's `BoundEffects`. Re-arms on every trigger fire.

**`Transfer`** (new terminal, replaces staged usage): one-shot adds an inner tree to a
target entity's `StagedEffects`. Consumed when triggered.

**`During(Condition, ...)`** (new wrapper): first-class state-scoped duration. Fires on
condition start, reverses on condition end. Conditions: `NodeActive`, `ShieldActive`,
`ComboActive(u32)`. Stays in `BoundEffects.conditions` permanently — condition monitors
activate/deactivate it each cycle. `During` with direct `Fire` requires a `Reversible`
effect. `During` with nested `When` relaxes to any effect.

**`SpawnedRegistry`** + **`EveryBolt`/`EveryCells`/etc.**: when a chip uses `Route(EveryBolt, ...)`,
the effect is split: existing bolts get it immediately (`ActiveBolts`), and a
`SpawnedRegistry` entry ensures every future bolt also receives it on spawn.

**`remove_source(entity, source_id)`**: new command that removes ALL entries from a given
source ID from an entity's `BoundEffects`, `StagedEffects`, and the `SpawnedRegistry`.
This enables chip unequip and protocol cleanup.

### New Trigger Vocabulary

**Local triggers** (fire on participant entities only):
`PerfectBumped`, `EarlyBumped`, `LateBumped`, `Bumped` (both bolt and breaker),
`Impacted(ImpactTarget)` (both entities), `Died` (victim only), `Killed(KillTarget)` (NEW — killer only)

Participant enums are per-trigger-concept:
`BumpTarget { Bolt, Breaker }`, `ImpactTarget { Impactor, Impactee }`,
`DeathTarget { Victim, Killer }`, `BoltLostTarget { Bolt, Breaker }`

**Global triggers** (fire on ALL entities with `BoundEffects`):
`PerfectBumpOccurred`, `EarlyBumpOccurred`, `LateBumpOccurred`, `BumpOccurred`,
`BumpWhiffOccurred`, `NoBumpOccurred`, `ImpactOccurred(ImpactTarget)`,
`DeathOccurred(DeathTarget)`, `BoltLostOccurred`, `NodeStartOccurred`, `NodeEndOccurred`,
`NodeTimerThresholdOccurred(f32)`, `TimeExpires(f32)`, `Spawned(EntityType)`

Note: `CellDestroyed` is replaced by `DeathOccurred(Cell)`. The `Killed(KillTarget)` trigger
is new — it fires on the entity that caused a kill (e.g., the bolt that destroyed a cell).

### New `walk_effects` Order: StagedEffects FIRST

The new dispatch helper walks `StagedEffects` before `BoundEffects`. This prevents a
single trigger from both arming and consuming a nested `When` in the same dispatch call.
Current code has the opposite order (BoundEffects first) — a behavioral difference in
edge cases involving same-trigger nesting.

### Bridge System Pattern (unchanged concept, cleaner API)

Each trigger has its own bridge system. The bridge calls `walk_effects(trigger, context,
entity, bound, staged, world)`. Bridges decide locality:
- Local: pass participant entities directly
- Global: query all entities with `BoundEffects`/`StagedEffects`

`walk_effects` takes a `&[Entity]` target list — caller decides scope.

Depth limit: `TriggerContext` carries a `depth: u32` counter. Dispatch bails at
`MAX_DISPATCH_DEPTH = 10` with a warning.

### New RON Format (from ron-migration examples)

```ron
// Deadline — global trigger (NodeTimerThresholdOccurred), fires on bolt (This)
Route(Bolt, When(NodeTimerThresholdOccurred(0.25), Fire(SpeedBoost(multiplier: 2.0))))

// Ricochet Protocol — local trigger (Impacted(Wall)), Until scoping
Route(Bolt, When(Impacted(Wall), Until(Impacted(Wall), Fire(DamageBoost(3.0)))))

// Transfer to impacted cell:
Route(Bolt, When(Impacted(Cell), On(ImpactTarget::Impactee, Transfer(When(Died, Fire(Explode(range: 60.0)))))))
```

## Part 3 — Integration Surface for Protocols

### Mapping Each Protocol to the New System

Of the 15 designed protocols, they split into three implementation categories:

#### Category A — Pure Effect Tree (use Route/Stamp/During/Transfer)

These protocols fully express as RON-authored effect trees in the new system. No custom
system needed beyond wiring them like chips.

| Protocol | New Effect Tree Pattern |
|---|---|
| **Deadline** | `Route(Bolt, When(NodeTimerThresholdOccurred(0.25), Until(NodeEndOccurred, Fire(SpeedBoost(2.0)))))` plus a parallel `DamageBoost`. Already exists as a chip — promote to protocol by marking `max_taken: 1` and source as protocol. |
| **Ricochet Protocol** | `Route(Bolt, When(Impacted(Wall), Until(Impacted(Wall), Fire(DamageBoost(3.0)))))` — already exists as chip, same. |
| **Fission** (every 8th cell destroyed → split bolt) | Requires a counter. Close to `EntropyEngine`-style counting. Could use a new `EffectKind::SplitBolt` + a bridge-side counter component on the bolt, OR express as a custom system that listens to `DeathOccurred(Cell)` and conditionally fires `SpawnBolts`. The counter logic cannot be expressed in the current tree. **Needs custom system.** |
| **Kickstart** | `Route(Bolt, Once(NodeStartOccurred, Until(Bumped, Fire(SpeedBoost(2.0)))))` plus parallel `DamageBoost` and `Piercing`. The "3 second duration" vs "first bump" gating needs clarification. If the window is bump-terminated, this is expressible. If it's time-terminated, requires `TimeExpires`. |
| **Debt Collector** | Complex: track bump-grade stack, apply multiplier on next cell impact. Requires a running counter component. Not expressible as a pure effect tree. **Needs custom system.** |

#### Category B — Custom System (game-rule modifiers, not effect trees)

These protocols modify game rules or maintain state that cannot be expressed as trigger →
effect chains. They read/write domain components or resources. They belong in
`src/protocol/systems/`.

| Protocol | Mechanism |
|---|---|
| **Tier Regression** | Modifies run progression state (tier tracking). Not effect system — pure state machine modification. |
| **Debt Collector** | Tracks per-bolt early/late bump stack, applies multiplier to next `DamageCell`. Custom system reading `BumpGraded` message, writing a `DebtStack(f32)` component on bolt. |
| **Iron Curtain** | On `BoltLost` message: spawns a directional damage wave entity. Close to effect — could be a new `EffectKind::DamageWave` — OR a custom system. Leaning toward custom because the wave spawning needs breaker-position data from the `BoltLost` context and the falloff behavior is bespoke. |
| **Echo Strike** | Tracks 3 most-recent echo cells (ring buffer on a resource/component). On perfect bump + cell impact: deals extra damage to echo cells. Complex stateful tracking. **Needs custom system.** |
| **Siphon** | Kill-streak timer + time accumulator. Listens to `DeathOccurred(Cell)`, tracks streak window with a `Timer`, adds to node timer resource. **Needs custom system.** |
| **Greed** | On chip offering skip: increments rarity boost counter. Pure UI/offering system interaction. **Needs custom system.** |
| **Reckless Dash** | Tracks dash contact zone (early/mid/risky); on risky contact applies damage multiplier to next cell impact. **Needs custom system.** |
| **Burnout** | Heat gauge (fills while moving, drains while still). Reads `BreakerVelocity` or input state each frame. **Needs custom system.** |
| **Anchor** | Already implemented as an `EffectKind::Anchor`. As a protocol it is promoted from evolution. The `Anchor` effect fires the existing system. Could be expressed as `Route(Breaker, Fire(Anchor { ... }))` in the new RON format — **pure effect tree** if the effect is already in `EffectType`. |
| **Conductor** | Changes which bolt is "primary" on perfect bump. Modifies `PrimaryBolt` marker and affects bolt-lost logic. **Needs custom system** — touches component marker insertion/removal. |
| **Afterimage** | On dash start: spawn phantom breaker entity at starting position. On perfect bump during phantom contact: insert `PhantomBolt` component on bolt. **Needs custom system** — entity spawning with position data. |
| **Fission** | See above. Custom system tracking per-run kill counter. |

#### Category C — Hybrid (effect tree + thin custom system bridge)

A few protocols are mostly expressible as effect trees but need a bridge system to provide
a trigger or context that doesn't exist yet.

| Protocol | What's Needed |
|---|---|
| **Ricochet Protocol** | Already purely an effect tree. No extra system needed. |
| **Kickstart** | Expressible if the "3s window" is terminated by `Bumped` rather than `TimeExpires`. If time-based, needs a bridge that fires `TimeExpires` in the right context. |
| **Iron Curtain** | Could become a new `EffectKind::DamageWave` with a bridge that translates `BoltLostOccurred` → fire the wave from breaker position. The position dependency makes a thin system cleaner. |

### `SourceId` for Protocol Cleanup

The new system's `SourceId` (a `String` key in `BoundEffects.sources`) is the mechanism
for protocol cleanup and conditional activation. Protocols should use a stable string
identifier as their `SourceId`:
- Convention: `"protocol:<protocol_name>"` e.g., `"protocol:deadline"`, `"protocol:ricochet"`
- `remove_source(entity, "protocol:ricochet")` removes all entries from that protocol from
  the entity's `BoundEffects`, `StagedEffects`, and `SpawnedRegistry`

**Protocol lifecycle via `SourceId`:**
- **Equip**: `stamp_effect(entity, "protocol:X", tree)` — routes tree into entity's `BoundEffects`
- **Cleanup on node end / run end**: `remove_source(entity, "protocol:X")` — removes all entries
- **Conductor / Afterimage "active state"**: these protocols modify component markers outside the
  effect tree, so `SourceId` only covers their effect-tree portions if any exist

### New Triggers Relevant to Protocols

The new system adds `Killed(KillTarget)` — fires on the killing entity when it destroys
something. This is directly useful for:
- **Siphon**: `Killed(Cell)` fires on the bolt; streak tracking can key off this
- **Fission**: `Killed(Cell)` fires on the bolt; counter increments on this trigger

Even though Siphon and Fission need custom systems (due to stateful counting), having
`Killed(Cell)` as a trigger lets them use the bridge infrastructure instead of listening
directly to raw game messages.

`Spawned(EntityType)` + `EveryBolt` routing is relevant for:
- **Fission**: newly split bolts should inherit parent's effects. `Route(EveryBolt, ...)` handles
  future bolts automatically via `SpawnedRegistry`.
- **Conductor**: when a new bolt is spawned (Fission split), it should start as non-primary;
  the `SpawnedRegistry` could stamp a "not primary" default.

### Summary: Protocol → System Category Map

| Protocol | Category | Hook |
|---|---|---|
| Deadline | A — effect tree | `Route(Bolt, When(NodeTimerThresholdOccurred(0.25), During(NodeActive, Fire(SpeedBoost + DamageBoost))))` |
| Ricochet Protocol | A — effect tree | `Route(Bolt, When(Impacted(Wall), Until(Impacted(Wall), Fire(DamageBoost(3.0)))))` |
| Anchor | A — effect tree | `Route(Breaker, Fire(Anchor { ... }))` |
| Kickstart | A or hybrid | `Route(Bolt, Once(NodeStartOccurred, Until(Bumped, Fire(...))))` |
| Iron Curtain | Hybrid | New `EffectKind::DamageWave` or thin custom system on `BoltLostOccurred` |
| Fission | B — custom system | Custom system on `Killed(Cell)` message + `SpawnBolts` effect |
| Debt Collector | B — custom system | Custom system on `BumpGraded` message + `DebtStack(f32)` component on bolt |
| Echo Strike | B — custom system | Custom system on perfect-bump + cell impact + `EchoBuffer([Cell; 3])` resource |
| Siphon | B — custom system | Custom system on `Killed(Cell)` / `DeathOccurred(Cell)` + streak timer |
| Greed | B — custom system | Hook into chip offering system, `GreedStacks(u32)` resource |
| Reckless Dash | B — custom system | Custom system monitoring dash phase + `RecklessReady(bool)` component |
| Burnout | B — custom system | `FixedUpdate` system reading breaker velocity + `HeatGauge(f32)` component |
| Conductor | B — custom system | Custom system on `BumpGraded(Perfect)` swapping `PrimaryBolt` marker |
| Afterimage | B — custom system | Custom system spawning phantom breaker on dash + bolt component swap |
| Tier Regression | B — custom system | Modifies tier/run state resource directly |

## Part 4 — Key Design Decisions for Protocol Implementation

### Do NOT target `src/effect/` for protocols

The planned refactor replaces `src/effect/` entirely. Protocols designed to the new system
(`Route`, `Stamp`, `Transfer`, `During`, `Killed`) will not need migration. Protocol
implementation should wait for, or be co-located with, the new effect system.

### Effect-tree protocols are chip-equivalent at the data level

Protocols that are pure effect trees (`Route(...)` definitions) are structurally identical
to chips. The only differences are:
- `max_taken: 1` (one per run)
- `SourceId` prefix `"protocol:"` (vs `"chip:"`) for identity tracking
- Offered via a different selection path (protocol slot, not chip slot)

This means the `ChipCatalog` / `dispatch_chip_effects` machinery (or its successor in the
new system) could serve protocol effects with minimal changes — a `ProtocolCatalog` resource
following the same `HashMap<String, ProtocolDefinition>` shape, where `ProtocolDefinition`
carries `effects: Vec<ValidDef>`.

### Custom-system protocols need a `ProtocolActive` guard

Every custom protocol system should gate on a resource/component indicating the protocol
is active for this run. The cleanest pattern is a `Resource` set of active protocols:
```rust
#[derive(Resource, Default)]
struct ActiveProtocols(HashSet<ProtocolKind>);
```
Each custom system adds `.run_if(resource_exists_and_equals...)` or queries for the
protocol's presence.

### `remove_source` covers effect-tree cleanup, not custom-system cleanup

For Category A protocols (effect tree), `remove_source(entity, "protocol:X")` cleans up
all `BoundEffects` / `StagedEffects` entries when a run ends. For Category B protocols
(custom systems), cleanup means despawning transient entities (phantom breaaker, echo
markers) and resetting per-run resources (`DebtStack`, `HeatGauge`, etc.) in
`OnExit(RunState::Node)` or `OnExit(GameState::Run)` systems.
